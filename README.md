# Dialogue Engine (DE-2.0)

Конструктор Telegram-ботов (фидбек-боты и постинг-боты) с рекламной
платформой, оплатой через ЮKassa и админ-панелью.

## Переменные окружения (.env)

```
MASTER_BOT_TOKEN=токен главного бота-конструктора
DATABASE_URL=postgresql+asyncpg://user:pass@host/db
SUPER_ADMIN_ID=твой telegram id (для доступа к админ-панели и модерации рекламы)

# ЮKassa (оплата рекламы) — обязательно, если пользуетесь /ads
YOOKASSA_SHOP_ID=...
YOOKASSA_SECRET_KEY=...
PAYMENT_CALLBACK_DOMAIN=https://dialogengine.ru
AD_WEBHOOK_PORT=8085
```

## Вебхук ЮKassa

Приложение поднимает отдельный aiohttp-сервер на `AD_WEBHOOK_PORT`
(по умолчанию 8085) со следующими путями:

- `POST /yookassa/webhook` — сюда ЮKassa шлёт уведомления об оплате
  (укажите `https://dialogengine.ru/yookassa/webhook` в личном кабинете
  ЮKassa как HTTP-уведомление). Nginx/traefik должен проксировать этот путь
  на порт `AD_WEBHOOK_PORT` контейнера.
- `GET /yookassa/return` — страница возврата после оплаты.

Обработчик вебхука **не доверяет** телу запроса — при получении уведомления
он переспрашивает статус платежа напрямую через API ЮKassa (`Payment.find_one`).

## Важно про перезапуск / деплой (TelegramConflictError)

Если видите в логах непрерывный `TelegramConflictError` дольше пары секунд —
это значит, что **два процесса одновременно держат getUpdates для одного и
того же токена бота**. Обычно это старый контейнер, который не успел
завершиться, когда поднялся новый (например, `docker compose up -d` без
ожидания остановки старого, или scale > 1 для одного и того же сервиса).

Приложение теперь умеет это обнаруживать: перед стартом поллинга каждого
дочернего бота процесс "застолбляет" его за собой в таблице
`bot_runtime_locks` (общая БД!) и обновляет метку каждые 10 секунд. Если
метка уже свежая и принадлежит другому процессу — новый процесс НЕ станет
запускать второй поллинг и просто напишет предупреждение в лог вида:

```
Bot <id> (@username) уже поднят ДРУГИМ процессом ...
```

Если видите такое предупреждение — значит, где-то реально висит второй живой
процесс с этой же БД. Проверьте:

- нет ли одновременно запущенного `docker-compose` на другом хосте/сервере с
  тем же `.env`;
- не запущен ли `main.py` локально для теста, пока на сервере уже крутится
  прод;
- `docker ps` — нет ли зависшего старого контейнера того же сервиса;
- используйте `docker compose up -d --force-recreate` вместо простого
  `restart`, и задайте достаточный `stop_grace_period`, чтобы старый процесс
  успел штатно завершиться (приложение само аккуратно останавливает всех
  дочерних ботов по SIGTERM).

Если зависший процесс не найден и предупреждение всё равно появляется —
удалите зависшую строку вручную: `DELETE FROM bot_runtime_locks WHERE bot_id=...`.

## Реклама (/ads)

В любом дочернем боте (фидбек или постинг) командой `/ads` можно купить:

1. **Показы в стартовых сообщениях** — фиксированный текст (до 100 символов)
   добавляется в конец `/start` во ВСЕХ ботах платформы, пока не закончится
   купленное число показов. Тариф считается по ступенчатой скидке за объём
   (см. `services/ads.py::price_for_impressions`).
2. **Разовая рассылка во все боты** — сообщение (с фото/видео и
   форматированием) рассылается всем пользователям всех активных ботов один
   раз. Ограничение: не чаще раза в 5 дней на одного покупателя.

Заявка уходит `SUPER_ADMIN_ID` в личку master-бота с кнопками
Принять/Отклонить. После одобрения покупателю приходит ссылка на оплату
ЮKassa; после успешной оплаты (через вебхук) реклама активируется
автоматически.

## Админ-панель (только SUPER_ADMIN_ID)

В главном меню master-бота появляется пункт «⚙️ Админ-панель»:

- **Все боты** — список абсолютно всех ботов платформы (не только своих), с
  полным доступом уровня владельца: вкл/выкл, настройки, статистика, кнопки.
- **Общая статистика** — сколько ботов всего, активных, по типам.
- **Разослать во все боты** — ручная рассылка произвольного сообщения (с
  медиа и форматированием) по всем пользователям всех активных ботов.

## Постинг-боты

- `/suggest` в постинг-боте временно переключает владельца/админа в обычный
  режим предложки — чтобы протестировать путь подписчика.
- Любое сообщение от админа/владельца создаёт черновик поста с кнопками:
  опубликовать сейчас, отложить (дата/время в формате `31.12.2026 18:30`,
  UTC), добавить inline-кнопки (`Текст | https://ссылка`, можно несколько),
  отменить.
- Отложенные посты публикует фоновый планировщик (`services/scheduler.py`,
  проверка раз в 20 секунд).
